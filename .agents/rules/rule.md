# エージェント・開発ルール (`calendit`)

このプロジェクトにおける AI エージェント（Antigravity）の開発・運用ルールを定義します。
## 0. ユーザーとのコミュニケーションルール
- implementation_planやコニュニケーションは**必ず日本語**

## 0. 概要
ターミナルから 各種カレンダーサービスを自在に操るための CLI ツール。人間のための Markdown 管理と、AI エージェントのための JSON 管理を両立します。
### 思想
以下の価値観を大切に開発します。
- 堅牢
- 安定
- 軽量
- 高速
- ユーザーフレンドリー
- システムフレンドリー
- デベロップフレンドリー

## 1. バージョン管理ルール
- **形式**: `YYYY-mmdd-[Sequence]`
  - 例: `2026-0412-01.06`
- **更新タイミング**: 実装の追加や重要な変更を行うたびに連動してインクリメントする。
- **ドキュメント更新**:
  - `spec/spec.md`: バージョン番号と最新仕様を反映。
  - `spec/history/YYYY-mmdd-XX.XX.md`: そのバージョンの詳細な変更内容を記録。

## 2. 実装・設計の原則
- **コンテキスト（用途）の尊重**: `--set <name>` オプションにより、カレンダーIDや認証情報を切り替えられるように実装する。
- **タイムゾーン**: ユーザーの入出力（Markdown）は、原則として PC（Mac）のローカル時刻を基準に扱う。
- **ID の保持**: Markdown 形式での書き出し時、予定の末尾に `(ID: ...)` を付与し、同期時のキーとして維持する。
- **中間サーバーの禁止**: 各カレンダーサービス（Google/Outlook）とローカルクライアントが直接通信する「自己完結型」を維持する。
- **型安全の徹底**: `as any` を新規に導入しない。設定値アクセスは型付きアクセサを用いて扱う。

## 3. 安全性と操作
- **破壊的操作の保護**: カレンダーの削除や `--sync` モードの実行前には、必ず `enquirer` を用いた確認プロンプト（Yes/No）を表示する。
- **Dry Run**: 変更を伴う操作には必ず `--dry-run` を用意し、影響範囲を事前確認できるようにする。

## 4. 自律的なワークフロー
- **ロードマップの監視**: `docs/development.md` の「ロードマップ」セクションを定期的に確認し、実装状況・優先度・未着手項目を同期する。
- **テスト駆動**: `docs/tests.md` に記載されたテストケースと期待動作に基づき、自律的に検証と修正を繰り返す。
- **自律的デバッグ・修正フロー**:
    1. テスト（`npm test`）を実行し、成否を確認。
    2. 失敗した場合、エラーログを解析し、原因となるソースコードの箇所（ロジック、型、環境差分等）を特定する。
    3. 修正案を考案・適用し、再度テストを実行。
    4. すべてのテストが「Success」になるまでこのループを継続し、完璧な品質を担保する。

## 5. 仕様書と履歴の整合性
- **転記の義務**: 議論され、実装要件に追加・変更された内容は、必ず以下のファイルに確実に転記すること。
    - `spec/spec.md`: 最新の全体仕様を反映。
    - `spec/history/YYYY-mmdd-XX.XX.md`: 各バージョンの変更詳細を記録。
- **不一致の禁止**: 「実装されているが仕様書にない」または「仕様書にあるが実装が古い」状態を放置しない。

## 6. 技術スタック
- **Runtime**: Node.js v18 (macOS native)
- **Language**: TypeScript (ESM)
- **CLI**: `commander`
- **Validation**: `zod`
- **Date/Time**: `date-fns-tz`
- **Prompt**: `enquirer`
- **Auth**: OAuth 2.0 (Google: tokens, Outlook: Keychain via MSAL)
- **Auth Library**: `@azure/msal-node`, `@azure/msal-node-extensions`

## 7. エラーハンドリング規約
- **カスタムエラーの必須化**: 新しい業務エラーは `src/core/errors.ts` の `CalendarError` 階層（`ConfigError` / `AuthError` / `ValidationError` / `ApiError`）を使用する。
- **ヒント付きエラー**: ユーザー起因で復旧可能なエラーには `hint` を付与し、次アクションを明示する。
- **終了処理の集約**: `process.exit()` をコマンド実装に直接書かず、`src/index.ts` の中央エラーハンドラに集約する。

## 8. ロギング規約
- **logger 利用の必須化**: ログ出力は `src/core/logger.ts` の `logger.debug/info/warn/error` を使用する。
- **console 直書きの禁止**: `console.log` / `console.warn` / `console.error` を新規実装で直接使用しない。
- **デバッグ手段の統一**: 詳細ログ確認は `--verbose` または `DEBUG=calendit` を用いる。

## 9. バリデーション規約
- **外部入力の検証**: 設定ファイル・CLI入力・ファイル入力など外部由来データは `zod` スキーマで検証する。
- **暗黙キャストの禁止**: 型安全性を崩す `as any` を使用しない。必要なら型を明示的に定義・拡張する。

## 10. テスト品質設計思想
### 基本方針（第三者検証品質を目標）
- **品質目標の明文化**: 「確かなテスト設計」「見落としのない品質検証」「仕様書とテストケースの確実な紐づけ」を必ず満たす。
- **トレーサビリティ義務**: `spec/spec.md` の全仕様セクションは、対応するテストケースを最低1件以上持つ。

### 適用するテスト技法
- **[EP] 同値分割法**: 有効クラス・無効クラスを識別して網羅する。
- **[BVA] 境界値分析**: 最小値・最大値・境界直前直後を重点検証する。
- **[DT] デシジョンテーブル**: 条件組み合わせを漏れなく検証する。
- **[ST] 状態遷移テスト**: 認証状態・設定状態・実行状態の遷移を追跡する。
- **[EG] エラー推測**: 仕様上の盲点、異常入力、運用時のヒューマンエラーを積極的に検証する。

### テストケース記述ルール
- **技法ラベルの付与**: テストケース ID に対応する技法ラベル（`[EP]` 等）を明示する。
- **仕様変更との同時更新**: 新仕様を実装したら、同時に `docs/tests.md` のケースとトレーサビリティ表を更新する。
- **データ独立性**: テストデータは `tests/data/` にケース単位で分離し、ケース間依存を避ける。
