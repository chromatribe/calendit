# calendit 徹底テスト用シナリオ集

本ドキュメントは、`calendit` のツールとしての機能性（Robustness）と、AIエージェントによる操作の妥当性（AI Behavior）の両面をブラッシュアップするためのテストシナリオをまとめたものです。

---

## 1. ツール・基礎機能テスト（Human-centric）
人間が直接、またはAIが基本的なツールとして利用する際の堅牢性を検証します。

| ID | カテゴリ | シナリオ内容 | 期待される結果 |
|---|---|---|---|
| F-1 | Markdown同期 | `query` で出した予定を書き換え `apply --sync` で反映 | 削除・変更・追加が完全一致でカレンダーに同期される |
| F-2 | 大量データ | 1ヶ月分（100件以上）の予定をCSVから一括登録 | バリデーションエラーなく、APIのレートリミット内で完了する |
| F-3 | 境界値 | 23:00 - 01:30 のような日付跨ぎ予定の登録・更新 | 日付を跨いでも正しく1つの予定として認識される |
| F-4 | 特殊文字 | 説明文に `#`, `*`, `[ ]`, 絵文字, URL 等を含む | `apply` と `query` を繰り返しても文字化けや崩れが発生しない |
| F-5 | 重複登録 | 同時刻に複数の予定を `add` または `apply` | 重複して登録される（または警告が出る）ことの確認 |

---

## 2. AIエージェント振る舞いテスト（Prompt-based）
ユーザーが自然言語でAIに依頼した際の「推論」と「操作」の正確性を検証します。

### 2.1 スケジューリングと推論
*   **シナリオ A-1: 曖昧なリスケ**
    *   **依頼**: 「来週月曜午後の『プロジェクト定例』を1時間後ろ倒しにして」
    *   **AI挙動**: 該当予定を特定し、時刻を加算して更新。
*   **シナリオ A-2: 横断空き時間検索**
    *   **依頼**: 「Google と Outlook 両方の来週水曜午後を見て、1時間空いてる所に『作業』と入れて」
    *   **AI挙動**: 複数コンテキストを `query` し、空き時間を算出して `add`。
*   **シナリオ A-3: セマンティック移動**
    *   **依頼**: 「Google にある『打ち合わせ』系の予定を全部『仕事用』コンテキストに移動させて」
    *   **AI挙動**: キーワードから対象を抽出し、Add + Delete で引っ越しを実行。

### 2.2 運用とエラー復帰
*   **シナリオ B-1: 安全確認 (Dry Run)**
    *   **依頼**: 「ファイルの内容で同期して。先に何が変わるか教えて」
    *   **AI挙動**: `--dry-run` の結果を解析し、人間に分かりやすく変更点を解説。
*   **シナリオ B-2: エラーからの自己修復**
    *   **依頼**: （トークン切れ状態で）「予定を入れて」
    *   **AI挙動**: `AuthError` を検知し、ユーザーに `auth login` を促した後、本来のタスクを続行。
*   **シナリオ B-3: レポーティング**
    *   **依頼**: 「先月、ジムに合計何時間行ったか計算して」
    *   **AI挙動**: 全予定から「ジム」を抽出し、Duration を合計して報告。

## 3. タスクリスト (Unified Todo) 連携テスト
Markdown 形式のタスク管理ファイルとカレンダーを同期させる高度なシナリオです。

### 3.1 記法に基づいた操作
*   **シナリオ T-1: デッドラインの可視化**
    *   **依頼**: 「タスクリストの LNW 領域で、来週月曜が締切のものをカレンダーに『[締切]』付きで入れて」
    *   **AI挙動**: Markdown を解析し、日付と本文を抽出して `add`。
*   **シナリオ T-2: P1タスクの作業枠確保**
    *   **依頼**: 「P1タスクのために、今日と明日の空き時間に1時間ずつ作業枠を予約して」
    *   **AI挙動**: 空きスロットを特定し、優先度順にカレンダーを埋める。
*   **シナリオ T-3: ステータス同期**
    *   **依頼**: 「[Done] になったタスクはカレンダーから消しておいて」
    *   **AI挙動**: 状態変化を検知し、対応するカレンダー ID を特定して `delete`。

### 3.2 大規模な変更
*   **シナリオ T-4: プロジェクトの一括リスケ**
    *   **依頼**: 「プロジェクト X が1週間遅れる。子タスクの日付を全部 +7日して、カレンダーもずらして」
    *   **AI挙動**: ファイル内の日付を一括置換し、カレンダー上の予定も連動して更新。
*   **シナリオ T-5: 未設定タスクの割り当て**
    *   **依頼**: 「日付が `[----/--/--]` のタスクを、来週の空き時間に1つずつ割り振って」
    *   **AI挙動**: 未定タスクをカレンダーの空き枠に配置し、ファイル側に日付を書き戻す。

## 4. macOS 固有テスト
macOS 標準カレンダーおよび常駐ブリッジの挙動を検証します。

*   **シナリオ M-1: ブリッジ安定性**
    *   `macos bridge start` 状態で数時間放置後の操作が正常か。
*   **シナリオ M-2: 権限（TCC）エラーの誘導**
    *   アクセス拒否された際に、システム設定での許可手順を正確に案内できるか。
*   **シナリオ M-3: 自動セットアップ**
    *   `macos setup` を使用して、対話的に正しいカレンダー ID をコンテキストに登録できるか。
